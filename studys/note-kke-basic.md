## 쿠버네티스와 카카오클라우드의 쿠버네티스 엔진 서비스의 기본 개념 정리
### MSA(Micro Service Architecture)의 등장과 가상화 기술
1. 소프트웨어 개발의 진화 과정<br>
- 모놀리식 아키텍쳐(MA): 모든 요소(코드)를 하나의 어플리케이션에 구축해서 배포하는 방식. 형태가 단순해서 유지보수가 편리하지만 부분 장애가 전체 서비스의 장애로 확대될 수 있다. 
- 서비스 지향 아키텍쳐(SOA): 서비스를 중심으로 한 방식, 느슨한 결합을 지향
- 마이크로 서비스 아키텍쳐(MSA)(현재): 각 서비스를 독립적으로 개발, 배포, 운영할 수 있는 구조
  -> 서비스가 모두 분산되어 복잡할 수도..

2. 가상화 기술<br>
= 실제 리소스를 추상화하고, 여러 개의 가상환경을 생성하는 기술

- 가상 머신(Virtual Machine, VM) : 하나의 물리적 서버에서 여러 개의 독립적인 가상 서버를 운영할 수 있도록 하는 기술
- 컨테이너 가상화(containerization) : 호스트 os 하나를 공유하고, 도커(Docker)와 같은 컨테이너 엔진이 서비스별 공간인 컨테이너를 만들어줌. os부터 응용 프로그램까지 모든 종속성을 컨테이너로 패키지화 하면 빠른 배포가 가능하다.

#### 가상화 구분
1. 하이퍼바이저를 통한 가상화
   - 베어메탈형: 컴퓨터 바로 위에 하이퍼바이저를 올림 = 클라우드 일반 vm 방식
   - 호스트형: 호스트 os 위에 하이퍼바이저를 설치 = 일반 사용자(로컬 pc) 대상<br>

     -> 하이퍼바이저 사용 유무에 따라 구분됨

### MSA와 컨테이너 가상화의 연관성

1. 독립적 배포 및 확장성

마이크로 서비스는 필요에 따라 개별적으로 확장/변경할 수 있는 환경이 필요하다. 컨테이너 기술은 각 서비스를 격리할 수 있기 때문에 유용.. 또한 이렇게 수많은 컨테이너가 잘 돌아가도록 감시하고, 관리하는게 쿠버네티스이다. = 컨테이너 오케스트레이션

2. 배포 및 유지보수 용이성

마이크로서비스 특성 상 배포와 롤백 등의 버전 관리가 잦은 편인데, 컨테이너 이미지나 가상머신 스냅샷을 사용하면 일관성 있고 빠른 배포가 가능하다.

### 쿠버네티스(Kubernetes) 개요
1. 등장배경 = 컨테이너 관리, 컨테이너 오케스트레이션의 중요성이 대두됨

2. 특징
- 선언적 배포: 개발자는 자신이 원하는 어플리케이션의 상태를 선언적으로 정의한다. 현재 상태를 관찰하고, 원하는 상태로 수렴시키는 과정을 반복
- 자동화된 스케일링: 어플리케이션의 부하에 따라 자동으로 리소스를 확장/축소함
  <img width="1510" height="540" alt="image" src="https://github.com/user-attachments/assets/44a8875e-78d0-4ab0-bc74-ad93524fa9c0" />
  (CA에서 노드 = 워커노드 로 이해하기)<br>
  -> CA는 클러스터 전체적인 물리적 자원이 부족할 때 사용되는 개념이다. 기존 워커 노드에 더 이상 새로운 파드(컨테이너)를 띄울 공간이 없다면, 새로운 워커노드가 자동으로 생성된다. 클러스터가 수용 가능한 워크로드를 최적화!
  
- 자가 치유: 실패한 컨테이너를 재시작하고, 워커 노드가 죽는 경우 응답하지 않는 컨테이너를 종료시켜서 클러스터의 안정성과 가용성을 유지한다.
  
  #### 관련 리소스
 - replica sets: 특정 파드의 복제본을 정의된 수만큼 유지되도록 관리하는 쿠버네티스 리소스
 - liveness probes: 파드 내 어플리케이션이 정상적으로 동작 중인지 확인하고, 문제가 발생하면 해당 컨테이너를 재시작함
 - readiness probes: 어플리케이션이 트래픽을 수용할 준비가 되었는지 확인하고, 준비되지 않은 파드는 서비스로 노출되지 않게 함

### 쿠버네티스 기본 개념
   
#### 파드란?<br>
= 최소 배포 단위, 하나 이상의 컨테이너로 이루어진다. 

<img width="320" height="220" alt="image" src="https://github.com/user-attachments/assets/8d00aa68-e850-493b-993e-43556eac89b1" />

#### 노드란?<br>
= 컨테이너화된 어플리케이션인 파드를 실행하는 서버, 워커 노드와 마스터 노드로 나뉜다. 

<img width="1000" height="350" alt="image" src="https://github.com/user-attachments/assets/aaa9e637-1f84-49db-a1ba-bfa4e9f674dc" />

- 마스터 노드 = 클러스터 운영을 위한 인프라가 모여서 함께 동작하는 노드, 클러스터 내 워커노드에게 작업을 할당한다. 
- 워커 노드 = 컨테이너가 실행되어서 실제로 작업이 수행되는 노드, 마스터 노드의 지시에 따라 컨테이너를 실행하고 관리한다.
- 
#### 클러스터란?<br>
= 여러 노드로 구성된 하나의 컴퓨팅 리소스 집합(kke에서 가장 먼저 생성)

### 쿠버네티스 오브젝트<br>
= 어플리케이션의 상태를 나타내는 엔티티로 특정 리소스를 뜻한다. 총 다섯 개가 있다. 

1. 파드
   - 독립적인 공간과 ip를 가지고, 동일한 워커 노드에서 실행되는 (서로다른) 컨테이너 간 네트워크와 스토리지는 공유된다.
2. 서비스
   - 파드를 통해 실행되고 있는 어플리케이션을 네트워크에 노출시키는 역할을 담당

#### 서비스가 필요한 이유<br>
-> 파드는 언제든 죽을 수 있는 일시적인 존재이다. 파드가 새로 생성될 때마다 내부 ip주소가 바뀌기 때문에, 다른 파드나 외부 사용자는 파드에 고정된 ip로 접근하는 것이 거의 불가능하다. 따라서 서비스는 여러 개의 파드를 그룹화하고, 이를 대표하는 고정된 ip(= 클러스터 ip)와 dns 이름을 노출한다.
